Issue #1 (CRITICAL): Audio Mode Initialize Before Component Renders
Current Location: artifacts/mobile/contexts/VoiceActivation.tsx Lines 65-76

Current Code (WRONG):

useEffect(() => {
  if (Platform.OS !== 'web') {
    import("expo-av").then(({ Audio }) => {
      Audio.setAudioModeAsync({
        allowsRecordingIOS: false,
        staysActiveInBackground: true,
        playsInSilentModeIOS: true,
        shouldDuckAndroid: true,
        playThroughEarpieceAndroid: false,
      }).catch((err: any) => {
        console.error('[VoiceContext] Audio init failed:', err);
      });
    }).catch((err: any) => {
      console.error('[VoiceContext] Expo-av import failed:', err);
    });
  }

  return () => {
    clearDismissTimeout();
    if (listenTimeoutRef.current) {
      clearTimeout(listenTimeoutRef.current);
      listenTimeoutRef.current = null;

Problem:

Audio init happens inside a dynamic import callback
Waits for expo-av to load (adds delay)
First TTS call might happen before audio mode is set
This is the #1 reason for "no sound on fresh install"
Fix: Move to synchronous import at top of file

Replace the imports section (top of file):

    clearDismissTimeout();
    if (listenTimeoutRef.current) {
      clearTimeout(listenTimeoutRef.current);
      listenTimeoutRef.current = null;
    }
  };
}, [clearDismissTimeout]);
Why: Guarantees audio mode is set before any TTS call.

Issue #3 (CRITICAL): Audio Mode Conflicts During Recording
Current Location: artifacts/mobile/services/speech.ts Lines 652, 689

Current Code (WRONG):

// Line 652 - During recording START:
async start(): Promise<void> {
  await Audio.setAudioModeAsync({
    allowsRecordingIOS: true,
    playsInSilentModeIOS: true,
  });
  // ... continue recording setup
}

// Line 689 - During recording STOP:
async stop(): Promise<URI | Blob> {
  await Audio.setAudioModeAsync({
    allowsRecordingIOS: false,
    playsInSilentModeIOS: true,
  });
  // ... continue cleanup
}
Problem:

Recording mode sets allowsRecordingIOS: true
TTS mode sets allowsRecordingIOS: false
They fight over audio mode
If TTS tries to play while recording mode is active, it fails silently
OR if leftover recording state persists after STT, TTS is blocked
Fix: Set ONE consistent audio mode at app startup (already done in Issue #1)

Then, remove these Audio.setAudioModeAsync calls from AudioRecorder and let the global init handle it:

// OLD (Lines 652-656):
async start(): Promise<void> {
  await Audio.setAudioModeAsync({
    allowsRecordingIOS: true,
    playsInSilentModeIOS: true,
  });
  // ...
}

// NEW - Remove the Audio.setAudioModeAsync, rely on global init:
async start(): Promise<void> {
  // Audio mode already set globally at app startup
  // Just start recording without changing mode
  // ...
}

// Similarly for stop() - remove the Audio.setAudioModeAsync call
Why: One authoritative audio mode (at startup) prevents conflicts.

Issue #4 (CRITICAL): Navigation Happens Before TTS Finishes
Current Location: artifacts/mobile/services/voiceRouter.ts Lines 479-567

Current Code (WRONG):

// Line 487 example (nav_home case):
speakText(msg, voice, CONFIRM_SPEED).catch((err) => {
  console.warn('[VoiceRouter] Navigation confirmation TTS failed:', err?.message);
});

Problem:

speakText() returns a Promise, but it's NOT awaited
.catch() only handles errors, doesn't wait for completion
Router navigates immediately while TTS is still playing
Audio context switches mid-playback → sound cuts off or never plays
Users hear no confirmation, just get navigated
Fix: Await the TTS before navigating

Pattern to replace everywhere (8 occurrences in lines 487-557):

// OLD:
speakText(msg, voice, CONFIRM_SPEED).catch((err) => {
  console.warn('[VoiceRouter] Navigation confirmation TTS failed:', err?.message);
});
router.replace("/student/home");

// NEW:
await speakText(msg, voice, CONFIRM_SPEED).catch((err) => {
  console.warn('[VoiceRouter] Navigation confirmation TTS failed:', err?.message);
  // Continue navigation even if TTS fails (don't reject)
});
router.replace("/student/home");
Important Notes:

The .catch() prevents rejection, so navigation continues even if TTS fails
This ensures UX never breaks, audio is just a bonus
Make sure executeGlobalNavigation() is marked async
